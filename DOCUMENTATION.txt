────────────────────────────────────────────────────────────────────────────────
  1. PROJECT OVERVIEW — WHAT IS CLEARDESK?
────────────────────────────────────────────────────────────────────────────────

ClearDesk is a full-stack IT support ticketing system. Here is what it does in
plain English:

  - A user fills in a form describing an IT problem (e.g. "My laptop won't
    turn on") and clicks Submit.
  - The app sends the ticket to Google's Gemini AI, which reads the problem and
    automatically decides:
      * What category it belongs to  (Hardware / Software / Network / Access / Other)
      * How urgent it is             (Low / Medium / High)
      * What the IT team should do   (a suggested resolution in plain text)
  - The ticket is saved permanently in a database.
  - All tickets appear in a dashboard table on the right side of the screen.
  - IT staff can filter tickets by category, urgency, or status, and update
    a ticket's status (Open → In Progress → Resolved) directly from the table.

The app is split into two halves:
  - BACKEND  (Python) — handles data, AI, and the database.
  - FRONTEND (HTML/CSS/JS) — what the user sees and interacts with in a browser.

They communicate via a REST API — the browser sends HTTP requests to Python,
and Python responds with JSON data.


────────────────────────────────────────────────────────────────────────────────
  2. HOW THE FILES CONNECT — THE BIG PICTURE
────────────────────────────────────────────────────────────────────────────────

When a ticket is submitted, data flows like this:

  [Browser form]
       │
       │  HTTP POST request (JSON)
       ▼
  [app.py — Flask web server]
       │
       ├──► [classifier.py] ──► [Google Gemini AI] ──► category + urgency + resolution
       │
       └──► [db.py] ──► [cleardesk.db — SQLite file] ──► ticket saved
       │
       │  HTTP 201 response (JSON)
       ▼
  [Browser — app.js updates the ticket table]

When the page loads or filters change:

  [Browser — app.js]
       │
       │  HTTP GET request
       ▼
  [app.py] ──► [db.py] ──► [cleardesk.db] ──► list of tickets
       │
       │  HTTP 200 response (JSON array)
       ▼
  [Browser — app.js renders the table]

File roles at a glance:

  backend/db.py         →  Talks to the SQLite database. Create, read, update.
  backend/classifier.py →  Talks to the Gemini AI API. Returns category/urgency.
  backend/app.py        →  The web server. Receives HTTP requests, calls db.py
                           and classifier.py, sends back responses.
  frontend/index.html   →  The HTML skeleton. Defines every element on the page.
  frontend/css/style.css→  The visual styling. Colours, layout, fonts, badges.
  frontend/js/app.js    →  The browser logic. Fetches data, renders the table,
                           handles form submission and filter clicks.


────────────────────────────────────────────────────────────────────────────────
  3. backend/db.py — THE DATABASE LAYER
────────────────────────────────────────────────────────────────────────────────

PURPOSE
-------
This file is the only place in the entire project that touches the database.
No other file imports sqlite3. This separation means: if you ever wanted to
switch from SQLite to a different database (like PostgreSQL), you only change
this one file and nothing else needs to know.

WHAT IS SQLITE?
---------------
SQLite is a database that stores all its data in a single file (cleardesk.db).
It is built into Python — no installation needed. It is perfect for small
applications and development.

IMPORTS
-------

  import os
    → The standard Python library for interacting with the operating system.
      Used here to read the DATABASE_PATH environment variable and to build
      the file path to cleardesk.db.

  import sqlite3
    → Python's built-in library for working with SQLite databases.
      Lets you run SQL commands (INSERT, SELECT, UPDATE) from Python.

  from contextlib import contextmanager
    → A tool from Python's standard library that lets you write a
      "context manager" — the thing that powers Python's "with" statement.
      Used here so database connections are always properly closed.

  from typing import Generator
    → A type hint helper. Generator[sqlite3.Connection, None, None] is a
      way of saying "this function yields a SQLite connection object".
      This is documentation for the code — Python doesn't enforce it at
      runtime, but it helps you understand what the function produces.

CONSTANTS / VARIABLES
---------------------

  _DB_PATH
    → A string storing the file path to the database file.
      The underscore prefix (_) is a Python convention meaning "private —
      not intended to be used outside this file".
      It first checks if an environment variable called DATABASE_PATH is set
      (useful for testing). If not, it defaults to cleardesk.db in the same
      folder as db.py.
      os.path.dirname(__file__) means "the folder this script lives in".
      os.path.join combines folder + filename into a full path.

  _CREATE_TICKETS_TABLE
    → A multi-line string containing a SQL statement that creates the
      tickets table if it doesn't already exist. This is called DDL
      (Data Definition Language).

      The table columns are:
        id                   — unique ticket identifier (e.g. "6F73C5D0")
        title                — short issue summary
        description          — full problem description
        submitter            — name of the person who submitted it
        category             — Network / Hardware / Software / Access / Other
        urgency              — Low / Medium / High
        status               — Open / In Progress / Resolved
        suggested_resolution — AI-generated fix suggestion
        created_at           — when the ticket was created (UTC timestamp)
        updated_at           — when it was last modified

      CHECK(...) constraints enforce that only allowed values can be stored.
      NOT NULL means the column cannot be empty.
      DEFAULT provides a fallback value if none is supplied.

FUNCTIONS
---------

  _get_conn()
    → A context manager (used with Python's "with" statement) that opens a
      connection to the database, yields it for use, then always closes it.

      Key settings applied to every connection:
        conn.row_factory = sqlite3.Row
          → Makes query results behave like dictionaries. Without this,
            rows are just tuples and you'd have to write row[0] instead of
            row["title"]. With it, you can write dict(row) to get a plain
            Python dictionary.
        PRAGMA journal_mode=WAL
          → Switches SQLite to Write-Ahead Logging mode. This allows
            multiple readers at the same time (important for a web server
            that handles concurrent requests).
        PRAGMA foreign_keys=ON
          → Tells SQLite to enforce foreign key relationships. Not used
            yet in this project but a good habit.

      The try/except/finally block:
        - If the code inside "with _get_conn() as conn:" succeeds → commit
          (permanently save the changes).
        - If an exception (error) is raised → rollback (undo all changes
          made during this connection so the database stays consistent).
        - finally → always close the connection, success or failure.

  init_db()
    → Creates the tickets table if it does not exist yet.
      Called once when Flask starts up. Safe to call multiple times because
      the SQL uses "CREATE TABLE IF NOT EXISTS".

  create_ticket(*, ticket_id, title, description, submitter, category,
                urgency, suggested_resolution, created_at, updated_at)
    → Inserts one new row into the tickets table and returns it as a dict.
      The * at the start of the parameters means ALL arguments must be
      passed by name (keyword-only), e.g.:
        create_ticket(ticket_id="ABC123", title="VPN broken", ...)
      This prevents accidentally passing arguments in the wrong order.

      Uses named SQL placeholders (:id, :title, etc.) instead of string
      formatting — this prevents SQL injection attacks.

      After inserting, it calls get_ticket() to fetch the row back so the
      returned data is exactly what was stored (SQLite constraints may
      normalise values).

  get_ticket(ticket_id)
    → Fetches a single ticket by its ID. Returns a dict or None if not found.
      ticket_id.upper() normalises the case so "abc123" and "ABC123" both work.

  list_tickets(*, category=None, urgency=None, status=None)
    → Returns all tickets, newest first, as a list of dicts.
      The three optional filter parameters let callers narrow the results:
        list_tickets(urgency="High")         → only high urgency tickets
        list_tickets(category="Network", status="Open") → AND filter
        list_tickets()                       → all tickets

      The WHERE clause is built dynamically: conditions and params are grown
      in parallel lists, then joined with AND. Using ? placeholders prevents
      SQL injection.

  update_ticket(ticket_id, fields)
    → Updates specific columns of one ticket. Only the columns present in
      the "fields" dictionary are changed; everything else stays the same.
      This is called a "partial update" or PATCH.

      SET clause is built dynamically using named placeholders (:col).
      cursor.rowcount tells us how many rows were affected — if 0, the
      ticket ID didn't exist and we return None.


────────────────────────────────────────────────────────────────────────────────
  4. backend/classifier.py — THE AI LAYER
────────────────────────────────────────────────────────────────────────────────

PURPOSE
-------
This file contains all the logic for communicating with Google's Gemini AI.
No other file imports google.genai. If you ever wanted to swap Gemini for a
different AI (like OpenAI's GPT), you would only change this file.

WHAT IS AN API?
---------------
An API (Application Programming Interface) is a way for two programs to talk
to each other. The Gemini API lets our Python code send text to Google's AI
and receive a response. We send a prompt (a question or instruction) and the
AI sends back an answer.

IMPORTS
-------

  import json
    → Python's built-in library for working with JSON data.
      JSON (JavaScript Object Notation) is a text format for structured data,
      e.g. {"category": "Network", "urgency": "High"}.
      json.loads() converts a JSON string into a Python dict.

  import re
    → Python's built-in library for regular expressions (regex).
      A regex is a pattern used to search or manipulate text.
      Used here to strip markdown formatting from AI responses.

  import time
    → Python's built-in library for time-related functions.
      time.sleep(3) pauses execution for 3 seconds — used in retry logic.

  from typing import TypedDict
    → A type hint tool. TypedDict lets you define a dictionary with known,
      named keys and their expected types. It is documentation — Python
      won't crash if you break it, but it makes the code much easier to
      understand.

  from google import genai
    → The Google Generative AI Python SDK (installed via pip).
      Provides the Client class for talking to Gemini.

  from google.genai import errors as genai_errors
    → The error types from the Gemini SDK. Used to catch specific failures:
        ClientError → 4xx HTTP errors (bad key, quota exceeded)
        ServerError → 5xx HTTP errors (Gemini is overloaded)

  from google.genai import types
    → Configuration objects for Gemini requests.
      types.GenerateContentConfig lets you set the system instruction
      and max_output_tokens per call.

CONSTANTS
---------

  VALID_CATEGORIES
    → A frozenset (an immutable, unordered set) of the 5 allowed categories.
      frozenset means it cannot be accidentally modified anywhere in the code.
      This is imported by app.py so there is ONE definition that both the
      AI classifier and the API validation use.

  VALID_URGENCIES
    → Same idea — the 3 allowed urgency levels.

  _SYSTEM_PROMPT
    → The instruction we give to Gemini before every ticket.
      It tells Gemini to act as an IT analyst and respond with raw JSON only.
      This is the "system message" — it sets the AI's behaviour for the
      entire conversation.

  _MODEL_NAME
    → The specific Gemini model to use: "gemini-2.5-flash".
      Gemini 2.5 Flash is fast and available on the free tier (5 requests
      per minute, 250,000 tokens per minute).

CLASSES
-------

  ClassificationResult (TypedDict)
    → Defines the shape of the dictionary returned by classify_ticket().
      Keys: category (str), urgency (str), suggested_resolution (str).
      Any code that calls classify_ticket() knows exactly what keys to
      expect in the returned dict.

  ClassificationError (Exception)
    → A custom exception class. When something goes wrong during
      classification (bad API key, AI sends garbage, etc.), we raise
      ClassificationError instead of a generic Python exception.

      __init__(self, message, raw_response="")
        → The constructor. Calls super().__init__(message) to initialise
          the parent Exception class with the error message.
          Also stores raw_response — whatever text the AI actually returned
          (even if it was unparseable). This is useful for debugging.

FUNCTIONS
---------

  _build_user_prompt(title, description)
    → Builds the text prompt sent to Gemini for each ticket.
      It embeds the ticket's title and description, and tells the AI exactly
      what keys to include in its JSON response and what values are allowed.
      Returns a plain string.

  make_client(api_key)
    → Creates and returns a genai.Client configured with the API key.
      Called once at startup in app.py. The client is reused for every
      request — creating it once is more efficient than creating it fresh
      every time a ticket arrives.

  _extract_json(text)
    → Tries three strategies to find valid JSON in the AI's response:
        Pass 1: Try json.loads(text) directly. Works if AI behaves correctly.
        Pass 2: Strip markdown code fences (```json ... ```) and try again.
                Gemini sometimes wraps its output in these even when told not to.
        Pass 3: Use regex to find the first {...} block anywhere in the text.
                Handles responses where the AI adds a sentence before the JSON.
      Raises ClassificationError if none of the three strategies work.

  classify_ticket(title, description, *, client)
    → The main public function. Sends the ticket to Gemini and returns a
      ClassificationResult dict.

      The * in the parameter list means "client" must be passed by name:
        classify_ticket("VPN broken", "Error 800", client=my_client)

      Retry logic:
        Gemini sometimes returns 503 (Service Unavailable) when demand is high.
        The function retries up to 3 times with exponential backoff:
          Attempt 1: runs immediately
          Attempt 2: waits 3 seconds
          Attempt 3: waits 6 seconds
          If all 3 fail: raises ClassificationError
        Client errors (4xx) like a bad API key are NOT retried — they would
        fail every time and retrying would just waste time.

      After a successful API call:
        1. response.text gives us the raw text the AI returned.
        2. _extract_json() parses it into a Python dict.
        3. We validate that category and urgency are in the allowed sets.
           If the AI "hallucinated" a value like "Critical" (not in our list),
           we raise ClassificationError rather than storing bad data.
        4. We return a ClassificationResult dict with the validated values.


────────────────────────────────────────────────────────────────────────────────
  5. backend/app.py — THE WEB SERVER
────────────────────────────────────────────────────────────────────────────────

PURPOSE
-------
This is the entry point of the backend. It starts a web server (Flask),
defines URL routes (what happens when the browser requests a specific URL),
validates incoming data, calls db.py and classifier.py, and sends back
JSON responses. It deliberately contains no SQL and no AI logic.

WHAT IS FLASK?
--------------
Flask is a Python web framework. A "web framework" is a library that handles
the plumbing of receiving HTTP requests and sending HTTP responses, so you can
focus on writing your application logic. Flask is deliberately minimal —
it gives you what you need without forcing you to use things you don't.

WHAT IS A REST API?
-------------------
REST (Representational State Transfer) is a convention for designing web APIs.
The key idea: URLs represent resources (e.g. /api/tickets is the "tickets"
resource) and HTTP methods describe what to do with them:
  GET     → Read / retrieve
  POST    → Create
  PATCH   → Partially update
  DELETE  → Delete (not used in this project)

WHAT IS JSON?
-------------
JSON is the data format used to communicate between the frontend and backend.
Example ticket as JSON:
  {
    "id": "6F73C5D0",
    "title": "Printer offline",
    "category": "Hardware",
    "urgency": "High",
    "status": "Open"
  }

IMPORTS
-------

  import logging
    → Python's built-in logging system. Lets us write messages to the
      terminal at different severity levels: DEBUG, INFO, WARNING, ERROR.
      Better than print() because log level can be controlled globally.

  import os
    → Used to read the GEMINI_API_KEY environment variable.

  import uuid
    → Python's built-in library for generating UUIDs (Universally Unique
      Identifiers). A UUID is a random 32-character hex string.
      We take the first 8 characters and uppercase them as the ticket ID.

  from datetime import datetime, timezone
    → datetime — represents a point in time.
      timezone — used to create timezone-aware datetimes.
      datetime.now(timezone.utc) gives the current UTC time.

  from dotenv import load_dotenv
    → Reads the .env file and loads its contents into os.environ.
      This means our code can call os.environ.get("GEMINI_API_KEY") and
      get the value we set in the .env file.

  from flask import Flask, jsonify, request
    → Flask     — the application class. app = Flask(__name__) creates the app.
    → jsonify   — converts a Python dict into a proper JSON HTTP response
                  with the correct Content-Type header.
    → request   — the current incoming HTTP request. Lets us read the
                  request body, query parameters, headers, etc.

  from flask_cors import CORS
    → CORS stands for Cross-Origin Resource Sharing. Browsers block requests
      from one origin (e.g. your HTML file) to a different origin (e.g. the
      Flask server on port 5000) by default for security. CORS(app) adds
      response headers that tell the browser to allow these requests.

  import db
    → Our own db.py module. Imported as a module (not specific functions)
      so call sites read clearly: db.create_ticket(...), db.list_tickets().

  from classifier import (VALID_CATEGORIES, VALID_URGENCIES,
                          ClassificationError, classify_ticket, make_client)
    → Imports specific names from our classifier.py module.

MODULE-LEVEL CODE (runs once when Flask starts)
-----------------------------------------------

  load_dotenv()
    → Reads backend/.env and puts GEMINI_API_KEY into the environment.

  logging.basicConfig(level=logging.INFO)
    → Configures the logging system to show INFO and above (INFO, WARNING,
      ERROR, CRITICAL) in the terminal. DEBUG messages are hidden.

  logger = logging.getLogger(__name__)
    → Creates a logger for this module. __name__ is "app" when running
      normally. Using a named logger means log messages show their source.

  app = Flask(__name__)
    → Creates the Flask application object. __name__ tells Flask where to
      find templates and static files.

  CORS(app)
    → Enables CORS for all routes. In production you would restrict this
      to your specific frontend domain.

  _api_key = os.environ.get("GEMINI_API_KEY")
    → Reads the API key from the environment. Uses .get() (not []) so it
      returns None instead of crashing if the key is missing.

  if not _api_key: raise RuntimeError(...)
    → If the key is missing, crash immediately with a clear error message
      rather than crashing later with a confusing error when the first
      ticket is submitted.

  _gemini_client = make_client(_api_key)
    → Creates the Gemini AI client once. Reused for every request.

  VALID_STATUSES = frozenset({"Open", "In Progress", "Resolved"})
    → Status lives in app.py (not classifier.py) because it is a workflow
      concept, not an AI output concept. The AI never sets status.

  with app.app_context(): db.init_db()
    → Runs init_db() inside Flask's application context (required for some
      Flask operations). Creates cleardesk.db and the tickets table if
      they don't exist. Safe to run on every server start.

UTILITY FUNCTIONS
-----------------

  utc_now_iso()
    → Returns the current UTC time as a string like "2026-04-15T08:33:24Z".
      The Z suffix is a standard notation meaning UTC (Zulu time).
      Uses datetime.now(timezone.utc) — the modern, recommended approach.

  make_ticket_id()
    → Generates a short ticket ID like "6F73C5D0".
      uuid.uuid4() generates a random UUID like "6f73c5d0-1234-...".
      [:8] takes the first 8 characters: "6f73c5d0".
      .upper() converts to uppercase: "6F73C5D0".

ROUTES (URL endpoints)
----------------------

  @app.route("/api/tickets", methods=["GET"])  →  list_tickets()
    → Returns all tickets as a JSON array, newest first.
      Accepts optional query parameters for filtering:
        ?category=Hardware    → only hardware tickets
        ?urgency=High         → only high urgency tickets
        ?status=Open          → only open tickets
        (combine multiple for AND logic)

      Validation: if a filter value is not in the allowed set (e.g.
      ?urgency=Critical), returns HTTP 400 with an error message.
      This gives the user clear feedback rather than a silent empty list.

  @app.route("/api/tickets", methods=["POST"])  →  create_ticket()
    → Creates a new ticket.
      Reads the JSON request body: { title, description, submitter }.
      title and description are required — returns 400 if either is missing.

      Calls classify_ticket() to get AI-assigned category, urgency, and
      suggested resolution. If the AI call fails (ClassificationError),
      it logs a warning and falls back to Other / Medium / "" — the ticket
      is STILL created. The AI is best-effort, not critical path.

      Returns the created ticket as JSON with HTTP 201 (Created).

  @app.route("/api/tickets/<ticket_id>", methods=["GET"])  →  get_ticket()
    → Returns a single ticket by its ID.
      <ticket_id> is a URL parameter — Flask extracts it and passes it as
      the ticket_id argument to the function.
      Returns 404 if the ID doesn't exist.

  @app.route("/api/tickets/<ticket_id>", methods=["PATCH"])  →  update_ticket()
    → Updates specific fields of an existing ticket.
      Reads the JSON body and validates each field against its allowed set.
      Only valid fields are passed to db.update_ticket() — invalid or
      unknown fields are silently ignored for forward compatibility.
      Always stamps updated_at with the current time.
      Returns 404 if the ticket doesn't exist.

ERROR HANDLERS
--------------

  @app.errorhandler(400), (404), (405), (500)
    → Flask by default returns HTML error pages. These handlers override
      that behaviour to always return JSON, even for errors.
      This is important because our frontend expects JSON responses and
      would fail to parse an HTML error page.
      The 500 handler logs the exception details for debugging.

STARTUP
-------

  if __name__ == "__main__": app.run(debug=True, port=5000)
    → Only runs the development server if this file is run directly
      (python app.py). Not executed when imported as a module.
      debug=True enables auto-reload (restarts when files change) and
      shows detailed error tracebacks in the terminal.
      port=5000 is the default Flask port.


────────────────────────────────────────────────────────────────────────────────
  6. frontend/index.html — THE PAGE STRUCTURE
────────────────────────────────────────────────────────────────────────────────

PURPOSE
-------
HTML (HyperText Markup Language) defines the structure and content of a web
page. Think of it as the skeleton — it says what elements exist on the page
and how they are nested, but not how they look (that is CSS) or how they
behave (that is JavaScript).

KEY ELEMENTS
------------

  <nav class="navbar">
    → The top navigation bar. Contains the "ClearDesk" brand name and the
      "Powered by Gemini" pill badge on the right.

  <div class="layout">
    → A container div that creates the two-column layout:
        Left column  → the sidebar (form)
        Right column → the main area (filter bar + table + detail panel)

  <aside class="sidebar">
    → The left panel. An <aside> is semantically an area of content that is
      related to but separate from the main content.

  <form id="ticket-form" novalidate>
    → The ticket submission form.
      novalidate disables the browser's built-in validation popups so we can
      show our own custom-styled error messages instead.
      id="ticket-form" lets JavaScript find and control this form.

  <div class="ai-thinking">
    → A loading banner that says "Gemini is analyzing your ticket…".
      Hidden by default via CSS. JavaScript adds "form--loading" to the
      form element which makes this banner visible via CSS rules.

  <div class="filter-bar" id="filter-bar">
    → Contains three groups of filter buttons: Category, Urgency, Status.
      data-filter-type="category" on each group tells JavaScript which
      filter dimension the buttons in that group control.
      data-value="Network" on each button tells JavaScript what value to
      filter by. An empty data-value means "All" (no filter).

  <table class="tickets-table">
    → The ticket dashboard table.
      <thead> contains the column headers (always visible).
      <tbody id="tickets-list"> is where JavaScript injects ticket rows.

  <div id="ticket-detail">
    → The detail panel below the table. Hidden by default. JavaScript
      adds "visible" class to show it when a ticket is clicked.
      aria-live="polite" tells screen readers to announce content changes.

  <div id="toast">
    → A notification popup (bottom-right). JavaScript sets its text and
      adds "show" class to make it appear. Auto-dismisses after 3 seconds.

  <script src="js/app.js">
    → Loads our JavaScript file. Placed at the end of <body> so the HTML
      elements exist before JavaScript tries to reference them.

ACCESSIBILITY ATTRIBUTES
------------------------

  aria-label       → Describes the element for screen readers.
  aria-live        → Tells screen readers to announce when content changes.
  aria-atomic      → Whether to announce the whole region or just the change.
  aria-hidden      → Hides decorative elements (like SVG icons) from screen readers.
  aria-required    → Tells screen readers this field must be filled.
  role="alert"     → Tells screen readers to announce this immediately (used
                     on validation error messages).
  tabindex="0"     → Makes a non-interactive element (like a <tr>) focusable
                     via keyboard Tab key.


────────────────────────────────────────────────────────────────────────────────
  7. frontend/css/style.css — THE VISUAL DESIGN
────────────────────────────────────────────────────────────────────────────────

PURPOSE
-------
CSS (Cascading Style Sheets) controls how HTML elements look. "Cascading" means
rules can override each other based on specificity and order.

KEY CONCEPTS USED
-----------------

  CSS Custom Properties (Design Tokens)
    → Defined in :root { --primary: #2563eb; ... }
      A custom property (variable) starts with --. You use it with var(--primary).
      Defined in :root they are available everywhere on the page.
      This creates a single place to change the entire colour scheme.

  Flexbox  (display: flex)
    → A layout mode where items are arranged in a row or column.
      The body uses flex with flex-direction: column so the navbar sits
      above the layout area.

  CSS Grid  (display: grid)
    → A layout mode for two-dimensional (row + column) arrangements.
      The .layout uses grid with grid-template-columns: 300px 1fr so the
      sidebar is a fixed 300px and the main area takes all remaining space.

  overflow-y: auto
    → If content is taller than the element, show a scrollbar. Without this,
      long content would push the layout beyond the screen.

  min-height: 0
    → A critical but counterintuitive CSS fix. When a flex child has overflow
      content, it can grow beyond its container by default. min-height: 0
      allows the child to shrink below its content size and respect overflow.

  CSS Classes for components:
    .badge         → Small coloured labels (category, urgency, status).
    .filter-btn    → The filter pill buttons above the table.
    .filter-btn.active → The currently selected filter button.
    .ticket-row    → A row in the ticket table.
    .ticket-row.selected → The highlighted row when a ticket is clicked.
    .empty-state   → The "no tickets" message shown when the table is empty.
    .ai-thinking   → The loading banner shown during AI classification.
    .input-error   → Red border on a form field with a validation error.
    .field-error   → The error message paragraph below a field.
    .toast         → The popup notification.
    .toast.show    → Makes the toast visible (transitions it in).
    .detail-*      → Styles for the ticket detail panel.

  Responsive design:
    @media (max-width: 768px) { ... }

  Transition / Animation:

      → Smoothly animates property changes (e.g. button hover colour change).

────────────────────────────────────────────────────────────────────────────────
  8. frontend/js/app.js — THE BROWSER LOGIC
────────────────────────────────────────────────────────────────────────────────

PURPOSE
-------
JavaScript runs in the browser and makes the page dynamic. Without JavaScript
the page would just be a static HTML file — clicking buttons wouldn't do
anything, the table would never update, and tickets couldn't be submitted.

This file is a "vanilla JS SPA" (Single Page Application):
  - Vanilla JS means no frameworks (no React, Vue, Angular).
  - SPA means the page never fully reloads — JavaScript updates parts of the
    page in response to user actions and API responses.

CONSTANTS / STATE VARIABLES
----------------------------

  const API_BASE = "http://localhost:5000/api"
    → The base URL for all backend API calls. Every fetch() call starts here.

  const form, submitBtn, ticketsList, ticketCount, detailPanel, filterBar, toastEl
    → References to HTML elements, cached with document.getElementById() once
      at load time. Caching is more efficient than searching the DOM every time.

  let selectedId = null
    → Tracks which ticket is currently selected (highlighted in the table
      and shown in the detail panel). null means nothing is selected.

  let toastTimer = null
    → Stores the ID of the setTimeout() that will hide the toast.
      Needed so we can clearTimeout() and reset it if the toast is shown
      again before it has finished hiding.

  const activeFilters = { category: null, urgency: null, status: null }
    → The current filter state. null means "show all" for that dimension.
      Persisted in this object so filters survive page operations like
      status updates (which reload the ticket list).

UTILITY FUNCTIONS
-----------------

  escHtml(str)
    → Converts special HTML characters to their safe equivalents:
        &  →  &amp;
        <  →  &lt;
        >  →  &gt;
        "  →  &quot;
        '  →  &#39;
      This prevents XSS (Cross-Site Scripting) attacks. If a malicious user
      submitted a ticket titled <script>stealYourCookies()</script>, without
      escaping it the browser would execute that script. With escaping, it
      is rendered as harmless visible text.

  formatDate(iso)
    → Converts an ISO timestamp like "2026-04-15T08:33:24Z" into a human-
      readable string like "Apr 15, 2026, 08:33 AM".
      new Date(iso) parses the string into a JavaScript Date object.
      .toLocaleString() formats it according to the user's local settings.

BADGE RENDERERS
---------------

  categoryBadge(category)
  urgencyBadge(urgency)
  statusBadge(status)
    → Each returns an HTML string like:
        <span class="badge badge-category-Hardware">Hardware</span>
      The CSS then colours each badge based on its class.
      statusBadge strips spaces from "In Progress" → "InProgress" because
      CSS class names cannot contain spaces.
      All three use escHtml() on the value to prevent XSS.

TOAST NOTIFICATIONS
-------------------

  showToast(message)
    → Displays a brief notification at the bottom-right of the screen.
      Sets the text, adds the "show" CSS class (which triggers a CSS
      transition to slide it in), and sets a 3-second timer to remove
      "show" (which triggers the slide-out transition).
      clearTimeout(toastTimer) cancels any previous timer so rapid calls
      don't cause the toast to disappear early.

FILTER BAR
----------

  filterBar.addEventListener("click", handler)
    → Uses event delegation: one listener on the parent (#filter-bar)
      instead of one listener per button. When any button is clicked, the
      event bubbles up to the filter bar.
      e.target.closest(".filter-btn") finds the nearest button ancestor
      of whatever was clicked (handles clicks on the button's text/icon).
      dataset.filterType reads data-filter-type from the .filter-btns div.
      dataset.value reads data-value from the clicked button.
      null for "All", a string value for a specific filter.
      After updating activeFilters, calls loadTickets() to refresh.

TICKET LIST FUNCTIONS
---------------------

  buildFilterQuery()
    → Builds a URL query string from the current activeFilters state.
      URLSearchParams is a browser built-in for building query strings safely.
      Result examples:
        All filters null → ""
        urgency=High     → "?urgency=High"
        urgency=High and status=Open → "?urgency=High&status=Open"

  loadTickets()
    → async function — can use await to pause and wait for network requests.
      Fetches the ticket list from the backend with current filters applied.
      fetch() is the browser's built-in HTTP request function.
      await pauses execution until the server responds.
      res.ok is true for 2xx responses. If not ok, throws an error.
      await res.json() parses the JSON response body into a JavaScript array.
      If anything fails (server down, network error), shows the error message.

  renderList(tickets)
    → Builds the table body HTML from the array of ticket objects.
      tickets.length → updates the ticket count in the header.
      Object.values(activeFilters).some(Boolean) → true if any filter is set
        (used to show different empty-state messages).
      If empty: renders a styled empty state with an SVG icon.
      If not empty: maps each ticket to an HTML <tr> string.
        template literals (backtick strings) allow multi-line HTML with
        JavaScript expressions embedded as ${...}.
        t.id === selectedId adds "selected" class to the current ticket.
        onclick="selectTicket(...)" calls the function when the row is clicked.
        event.stopPropagation() on the status cell prevents the row click
        from firing when the user changes the dropdown — otherwise clicking
        the dropdown would also open the detail panel.

TICKET DETAIL PANEL
-------------------

  selectTicket(id)
    → async function called when a ticket row is clicked.
      Immediately updates selectedId and applies the "selected" CSS class to
      the correct row (before the network request completes) so the UI feels
      instant.
      Then fetches the full ticket details from GET /api/tickets/:id.
      Calls renderDetail() to populate the panel.

  renderDetail(t)
    → Builds the HTML for the detail panel from a ticket object t.
      Adds the "visible" CSS class to the panel element to show it.
      Renders the ticket's title, submitter, timestamps, all three badges,
      the full description, and the AI suggested resolution (in a blue box).
      Includes a status <select> dropdown so the status can be changed here
      too (not just in the table row).

STATUS UPDATES
--------------

  updateStatus(id, status)
    → async function called when a status dropdown is changed.
      Sends a PATCH request to /api/tickets/:id with { status: "..." }.
      fetch() with method: "PATCH" and a JSON body.
      JSON.stringify() converts the JavaScript object to a JSON string.
      On success: shows a toast, reloads the ticket list, and if the changed
      ticket's detail panel is open, re-fetches it to keep badges in sync.

FORM VALIDATION
---------------

  showFieldError(field, message)
    → Adds the "input-error" CSS class to the field (red border).
      Creates a <p> element with class "field-error" and injects it below
      the input. Checks for an existing error first to avoid duplicates.
      role="alert" on the <p> makes screen readers announce the error.

  clearFieldError(field)
    → Removes "input-error" class and deletes the error <p> element.
      Called when the user starts typing in the field.

FORM SUBMISSION
---------------

  form.addEventListener("submit", async handler)
    → e.preventDefault() stops the browser from doing its default form
      submission (which would reload the page).
      Reads and trims values from the three input fields.
      Validates title and description — shows field errors and returns early
      if either is empty.
      Adds "form--loading" class → CSS dims inputs and shows the banner.
      Disables all form inputs and the submit button to prevent double-submit.
      Sends POST request with JSON body { title, description, submitter }.
      On success: resets the form, clears errors, shows toast, reloads list,
        opens the new ticket's detail panel.
      On failure: shows the error message in a toast.
      finally block: always re-enables the form regardless of success/failure.

EVENT LISTENERS FOR VALIDATION CLEARING
----------------------------------------

  ["title", "description"].forEach(id => ...)
    → Loops over the two required field IDs and attaches an "input" event
      listener to each. "input" fires on every keystroke.
      When the user types in a field, clearFieldError() is called immediately
      so the error disappears as they fix it.

BOOTSTRAP
---------

  loadTickets()
    → Called once at the bottom of the file — runs when the page loads.
      Immediately fetches and renders the current tickets from the database.

────────────────────────────────────────────────────────────────────────────────
  10. KEY CONCEPTS GLOSSARY
────────────────────────────────────────────────────────────────────────────────

API (Application Programming Interface)
  A defined way for two programs to communicate. Our browser talks to Flask
  via a REST API using HTTP requests and JSON responses.

async / await (JavaScript)
  async marks a function as asynchronous — it can pause and wait without
  blocking the browser. await pauses until a Promise (e.g. a fetch() call)
  resolves. This lets the browser stay responsive while waiting for the server.

CORS (Cross-Origin Resource Sharing)
  A browser security rule that blocks JavaScript from making requests to a
  different domain/port than the page it came from. Flask-CORS adds response
  headers that tell the browser it is allowed.

CSS Class
  A reusable style identifier. class="badge badge-urgency-High" applies two
  classes. CSS rules targeting .badge-urgency-High set that element's colours.

DOM (Document Object Model)
  The browser's in-memory representation of the HTML page as a tree of objects.
  JavaScript can read and modify the DOM: document.getElementById("title")
  finds the element with id="title" and returns a JavaScript object you can
  manipulate.

Event Delegation
  Attaching one event listener to a parent element instead of many listeners
  to child elements. When a child is clicked, the event "bubbles up" to the
  parent. More efficient and works even for elements added dynamically.

Exception (Python) / Error (JavaScript)
  An object representing something that went wrong. In Python, you "raise" an
  exception and "catch" it with try/except. In JavaScript, you "throw" an
  error and "catch" it with try/catch.

fetch() (JavaScript)
  The browser's built-in function for making HTTP requests. Returns a Promise.
  Used to GET tickets from the backend and POST/PATCH ticket data.

frozenset (Python)
  An immutable set — like a set but it cannot be modified after creation.
  Used for VALID_CATEGORIES and VALID_URGENCIES to ensure they are never
  accidentally changed by other code.

HTTP Status Codes
  Numbers indicating the result of an HTTP request:
    200 OK          → Success (GET, PATCH).
    201 Created     → Success, a new resource was created (POST).
    400 Bad Request → The client sent invalid data.
    404 Not Found   → The requested resource doesn't exist.
    405 Method Not Allowed → Wrong HTTP method for this URL.
    429 Too Many Requests → Rate limit exceeded (too many API calls).
    500 Internal Server Error → Something crashed on the server.
    503 Service Unavailable → Server is overloaded or down temporarily.

JSON (JavaScript Object Notation)
  A text format for structured data. Uses {} for objects, [] for arrays,
  strings in double quotes. Both Python (json module) and JavaScript
  (JSON.parse / JSON.stringify) can read and write it natively.

Keyword-only arguments (Python)
  Parameters after a bare * in a function signature must be passed by name:
    def create_ticket(*, ticket_id, title, ...)
    create_ticket(ticket_id="ABC", title="VPN broken")  ← correct
    create_ticket("ABC", "VPN broken")                  ← TypeError

REST (Representational State Transfer)
  A design style for web APIs. Resources (like tickets) are identified by
  URLs, and HTTP methods (GET, POST, PATCH, DELETE) describe actions.

SQL (Structured Query Language)
  The language used to talk to relational databases like SQLite.
  Key statements used in this project:
    CREATE TABLE  → Define the schema (structure of the data).
    INSERT INTO   → Add a new row.
    SELECT        → Read rows.
    UPDATE        → Modify existing rows.
    DELETE        → Remove rows (used manually to clear tickets).

SQL Injection
  An attack where a malicious user includes SQL code in their input, hoping
  the server will execute it. Prevented by using parameterised queries
  (? or :name placeholders) instead of building SQL strings with f-strings.

TypedDict (Python)
  A way to define a dictionary with a known set of keys and their types.
  ClassificationResult is a TypedDict — it tells other code exactly what
  keys to expect when classify_ticket() returns a result.

UUID (Universally Unique Identifier)
  A randomly generated 128-bit number, formatted as 32 hex characters.
  Used as the basis for ticket IDs. The probability of two UUIDs colliding
  is astronomically low.

Virtual Environment (venv)
  An isolated Python installation for a project. Packages installed inside
  the venv don't affect other Python projects on your machine.
  Created with: python -m venv venv
  Activated with: venv\Scripts\activate (Windows)

XSS (Cross-Site Scripting)
  A security attack where malicious JavaScript is injected into a web page.
  Prevented in this project by running all user-supplied content through
  escHtml() before inserting it into the page with innerHTML.
